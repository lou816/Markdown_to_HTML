# Markdown to HTML Converter

> 一個即時轉換 Markdown 為 HTML 預覽的網頁應用程式

---

## 專案簡介

### 你如何理解題目需求、規劃專案結構

#### 題目需求理解

本專案是 Markdown 即時預覽編輯器，核心需求：

1. **技術框架**：React + Vite
2. **版面配置**：左右分欄（左側編輯，右側預覽）
3. **Markdown 支援**：標題、粗體、斜體、程式碼區塊、超連結
4. **UI 優化**：使用 npm 套件（Tailwind CSS）

#### 專案規劃

**技術選型**：

| 項目 | 選擇 | 理由 |
|------|------|------|
| 建構工具 | Vite | 快速開發伺服器、輕量化 |
| UI 框架 | Tailwind CSS | 快速開發、檔案體積小 |
| Markdown 解析 | marked | 輕量、API 簡單 |
| 程式碼高亮 | highlight.js | 語言支援豐富 |

**元件架構**：

```
src/
├── components/
│   ├── Header/              # 頂部導航列
│   ├── Layout/              # 版面配置（管理狀態）
│   ├── MarkdownEditor/      # Markdown 輸入區
│   └── HtmlPreview/         # HTML 預覽區
└── utils/
    └── markdownParser.js    # Markdown 轉換邏輯
```

**設計原則**：
- 單一職責：每個元件只負責一項功能
- 單向資料流：Layout 管理狀態，傳遞給子元件
- 左右並排布局：編輯區和預覽區 50/50 分割

## AI 回覆的品質與你的判斷

### 哪些 AI 回覆幫助最大？

1. **專案架構設計**
   - 提出元件化架構（Header、Layout、Editor、Preview）
   - 說明單一職責原則與關注點分離
   - 符合 React 最佳實踐

2. **技術選型建議**
   - 提供多種方案比較（marked vs react-markdown）
   - 說明各方案優缺點
   - 幫助做出適合專案規模的決策

3. **開發規範建立**
   - 自動生成完整的設計文件模板
   - 定義清晰的介面規範
   - 便於後續實作

### 哪些回覆錯誤或不實用？

1. **XSS 防護方案不夠完整**
   - **AI 的限制**：初期建議只用自訂 `escapeHtml()` + 限制 renderer 方式防護
   - **問題**：無法應對複雜的 HTML 注入攻擊，防護不夠系統化
   - **解決方式**：參考已知的安全最佳實踐（DOMPurify 多層防護機制），自行實現：
     - escapeHtml + URL 驗證 + DOMPurify 清理
     - 設定白名單（允許的標籤、屬性、URL 協議）
     - 禁止危險事件屬性（onerror、onload、onclick 等）

### 開發流程與 AI 的角色

**採用 TDD 開發流程**：
1. 撰寫設計文件（claude.md）定義需求、Props、State、UI 規範、測試案例
2. 撰寫測試用例（test.md）
3. 要求 AI 根據設計與測試實現代碼
4. 驗證實現是否通過所有測試

## 在文件中分析「AI 生成程式碼的優缺點」

### 優點 ✅

1. **大幅提升開發速度**
   - 本專案初始化到架構設計僅 30 分鐘
   - 重複性工作效率提升約 80%

2. **提供最佳實踐**
   - 自動套用業界標準架構
   - 即時解答技術問題

3. **降低技術門檻**
   - 快速產生可運作的原型

### 缺點 ⚠️

1. **程式碼品質需人工審查**
   - 可能使用過時 API
   - 需透過 Linter 統一風格

2. **安全性需特別注意**
   - 可能未處理 XSS 等漏洞

3. **缺乏深度理解**
   - 無法真正理解業務邏輯
   - 創新架構仍需人類設計

4. **可能降低學習動機**
   - 過度依賴會不理解原理

### 最佳實踐

**人類應負責的部分**：
- 撰寫詳細的設計文件和測試用例
- 進行安全性審查和性能測試
- 定義需求、架構和驗收標準

**AI 應負責的部分**：
- 根據設計文件和測試實現代碼
- 確保實現通過所有測試用例
- 生成樣板代碼和配置文件

> **核心原則**：人類設計 + 測試 → AI 實現 → 驗證通過。AI 是工具，應在明確的設計和測試框架下使用。
